# Лабораторная работа labyrinths

## Описание работы
Необходимо разработать консольное приложение на Java, которое находит кратчайшие пути в лабиринтах.

Программа должна находить путь от стартовой точки S до конечной точки F и сохранять результат в файл.


## Требования

### Функциональные требования
1) Должны быть реализованы хотя бы два алгоритма - алгоритмы [Дейкстры](https://neerc.ifmo.ru/wiki/index.php?title=%D0%90%D0%BB%D0%B3%D0%BE%D1%80%D0%B8%D1%82%D0%BC_%D0%94%D0%B5%D0%B9%D0%BA%D1%81%D1%82%D1%80%D1%8B) и [A*](https://neerc.ifmo.ru/wiki/index.php?title=%D0%90%D0%BB%D0%B3%D0%BE%D1%80%D0%B8%D1%82%D0%BC_A*&mobileaction=toggle_view_mobile)
2) Сохранение решения в файл
3) Ввод конфигурации лабиринта, начальной и конечной точки через CLI
4) Путь должен быть кратчайшим. В случае существования нескольких кратчайших путей подходит любой.
5) Доступные перемещения на пути - влево, вправо, вниз, вверх. Не ожидаются перемещения по диагонали, сквозь стены и т.п.
6) Если набор аргументов CLI верный, то файл должен создаваться в любом случае, но в случае ошибки должен быть оставлен пустым.

### Нефункциональные требования
1) Используйте наследование и абстракции (в том числе, обратите внимание на схожесть алгоритмов A* и Дейкстра)
2) Должны соблюдаться принципы SOLID. Архитектура решения должна позволять добавление новых алгоритмов без изменения существующего кода. 
3) Проект должен быть структурирован по пакетам
4) Программа должна корректно обрабатывать ошибки ввода (некорректный формат лабиринта, отсутствие пути, неверные аргументы CLI и т.п.) и выводить понятное сообщение пользователю.

### Ограничения
1) Позволяется использовать все возможности **_стандартной библиотеки Java_**
2) Не дозволяется использовать сторонние зависимости (кроме уже включённых, если таковые есть в [pom.xml](pom.xml))


## Режимы работы
Реализация других вариантов наборов аргументов допустима, но необязательна. Количество аргументов для приведённых вариантов должно строго совпадать с описанием.
### 1. Без аргументов
При запуске без аргументов программа должна вывести список доступных алгоритмов:

Пример:
```
Available solvers:
- dijkstra
- astar
```

### 2. Решение лабиринта
_\<algorithm\> \<input_file\> \<output_file\> \<height\> \<width\>_

* _height_ - корректная высота лабиринта _(см. пункт формат лабиринта)_
* _width_ - корректная ширина лабиринта _(см. пункт формат лабиринта)_
* _input_file_ - путь к входному файлу лабиринта
* _output_file_ - путь к выходному файлу с решением
* _algorithm_ - алгоритм для решения

Пример:
```
astar maze.txt solved.txt 20 30
```

_Для обязательных алгоритмов названия должны совпадать с примером с точностью до регистра_

## Формат лабиринта
### Входной файл:
* '\#' — стена
* ' ' (пробел) — проход
* 'S'— стартовая точка
* 'F' — конечная точка

Корректный формат лабиринта предусматривает ограничение рамкой (первая и последняя строка, первый и последний столбец - всегда состоят из \#)

В лабиринте должны быть ровно одна S и ровно одна F

Лабиринт строго прямоугольный

Пример корректного входного файла:
```
#####################
#S          #   #   #
# ####### ##### # # #
#     #   #     # # #
##### # ### ##### # #
# #   #   # #     # #
# # ##### # ##### # #
# #     # #       # #
# ##### # ######### #
#       #F    #   # #
# ######### # # ### #
#     #     #   #   #
# ##### ####### # ###
# #   # # #   # # # #
### # # # # # # # # #
#   # # #   #   # # #
# ### # ##### ### # #
#   #   #   #   # # #
# # ##### # ##### # #
# #       #         #
#####################
```


### Выходной файл:
* '\#' — стена
* ' ' (пробел) — проход
* 'S' — стартовая точка
* 'F' — конечная точка
* '.' — путь

_Расположение стартовой и конечной точек должны совпадать с исходным лабиринтом_

Пример корректного выходного файла:
